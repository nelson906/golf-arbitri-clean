🔍 Analisi critica (thinking process)

Il problema principale del codebase è la proliferazione di stringhe magiche per concetti di dominio fondamentali: user_type, level, status, role. Queste stringhe compaiono in oltre 30 file diversi e non sono type-safe. PHP 8.1+ ha gli Enum nativi proprio per questo — e il refactoring si ammortizerà con il primo typo evitato in produzione.

Il secondo tema è la duplicazione della logica di visibilità: lo stesso switch su user_type è replicato nel Trait HasZoneVisibility, nel scope Tournament::scopeVisible(), nel scope User::scopeVisible(), e sporadicamente anche in metodi privati di controller. Ogni nuova regola di visibilità deve essere aggiornata in 4+ posti.

Il terzo tema è il NotificationController::sendNationalNotification() che è un metodo da 200 righe che fa tutto: validazione, costruzione destinatari, invio email, aggiornamento DB. Il metodo resendNationalNotification() duplica il 70% dello stesso codice.

1. PHP Enums — Le stringhe magiche del dominio

Critico Magic strings per concetti di dominio — 30+ file coinvolti

I concetti chiave del dominio (user_type, level, role assignment, status torneo) sono rappresentati come stringhe disperse nell'intero codebase. Un refactoring con PHP 8.1+ Backed Enums porta type-safety, autocompletamento IDE e un unico source of truth.

2. Duplicazione logica visibilità (Trait vs Scope)

Critico La stessa logica di visibilità esiste in 4 posti diversi

La logica di visibilità basata su user_type è implementata in:

  • HasZoneVisibility::applyTournamentVisibility()
  • Tournament::scopeVisible()
  • User::scopeVisible()
  • Metodi privati nei controller (es. getAvailableReferees())
⚠️
Quando cambiano le regole di accesso (es. un nuovo ruolo) devi modificare 4+ posti. La soluzione è far sì che gli Scope del Model usino il Trait (o viceversa) come singolo source of truth.

Soluzione: il Trait diventa la fonte, gli Scope delegano

✅ Soluzione preferita — app/Support/TournamentVisibility.php
namespace App\Support;

use App\Enums\UserType;
use App\Models\User;
use Illuminate\Database\Eloquent\Builder;

final class TournamentVisibility
{
    public static function apply(
        Builder $query,
        ?User $user = null
    ): Builder {
        $user = $user ?? auth()->user();

        if (! $user) {
            return $query->whereRaw('1 = 0');
        }

        $type = UserType::from($user->user_type);

        return match (true) {
            $type->seesEverything()
                => $query,
            $type === UserType::NationalAdmin
                => $query->whereHas('tournamentType',
                    fn ($q) => $q->where('is_national', true)),
            $type === UserType::ZoneAdmin
                => $query->whereHas('club',
                    fn ($q) => $q->where('zone_id', $user->zone_id)),
            $type === UserType::Referee
                => self::applyRefereeVisibility($query, $user),
            default
                => $query,
        };
    }

    private static function applyRefereeVisibility(
        Builder $query,
        User $user
    ): Builder {
        $level = RefereeLevel::tryFrom($user->level ?? '');
        $isNational = $level?->isNational() ?? false;

        if ($isNational && $user->zone_id) {
            return $query->where(function ($q) use ($user) {
                $q->whereHas('club',
                    fn ($s) => $s->where('zone_id', $user->zone_id))
                  ->orWhereHas('tournamentType',
                    fn ($s) => $s->where('is_national', true));
            });
        }

        return $user->zone_id
            ? $query->whereHas('club',
                fn ($q) => $q->where('zone_id', $user->zone_id))
            : $query->whereRaw('1 = 0');
    }
}
💡 Perché questa scelta

Una classe final statica è il posto migliore per logica di visibilità: non ha stato, non ha dipendenze dal container, è testabile in isolamento e può essere usata sia dai Trait che dagli Scope Eloquent senza duplicazione. Il pattern match(true) con le guardie booleane è più leggibile di un chain di if/elseif e sfrutta il PHP 8.0+ match exhaustiveness.

3. NotificationController — God Method da 200 righe

Critico sendNationalNotification() e resendNationalNotification() — 70% di codice duplicato

I metodi sendNationalNotification() e resendNationalNotification() fanno entrambi:

  1. Costruzione manuale dei destinatari (TO + CC) con query Eloquent inline
  2. Invio email con Mail::raw() in un loop foreach
  3. Logica di fallback (se TO vuoto, usa il primo CC come TO)
  4. Aggiornamento del record TournamentNotification
  5. Costruzione del messaggio di risposta
🚨
Usa Mail::raw() — email senza template HTML, senza Mailable class, senza testabilità. Il contenuto email è costruito inline nel controller.

Soluzione: RecipientBuilder + Mailable + Action class

namespace App\Services;

use App\Enums\AssignmentRole;
use App\Models\Tournament;
use App\Models\User;

final class NotificationRecipientBuilder
{
    /** @var array<array{email: string, name: string}> */
    private array $to = [];
    private array $cc = [];

    public function addCampionati(): static
    {
        $this->to[] = [
            'email' => config('golf.emails.ufficio_campionati'),
            'name'  => 'Comitato Campionati',
        ];
        return $this;
    }

    public function addZone(Tournament $tournament): static
    {
        $zone = $tournament->club?->zone;
        if ($zone?->email) {
            $this->cc[] = ['email' => $zone->email, 'name' => $zone->name];
        }
        return $this;
    }

    public function addZoneAdmins(Tournament $tournament): static
    {
        User::where('user_type', 'admin')
            ->where('zone_id', $tournament->club?->zone_id)
            ->where('is_active', true)
            ->each(fn ($u) => $this->cc[] = [
                'email' => $u->email, 'name' => $u->name
            ]);
        return $this;
    }

    public function addAssignedReferees(Tournament $tournament): static
    {
        $tournament->assignments->each(function ($a) {
            if ($a->user) {
                $this->cc[] = ['email' => $a->user->email, 'name' => $a->user->name];
            }
        });
        return $this;
    }

    public function addObservers(Tournament $tournament): static
    {
        $tournament->assignments
            ->where('role', AssignmentRole::Observer->value)
            ->each(function ($a) {
                if ($a->user) {
                    $this->cc[] = ['email' => $a->user->email, 'name' => $a->user->name];
                }
            });
        return $this;
    }

    /** @return array{to: array, cc: array} */
    public function build(): array
    {
        return ['to' => $this->to, 'cc' => $this->cc];
    }
}

5. Tournament Model — L'accessor zone_id e la doppia fonte di verità

Alta zone_id esiste nel DB e viene anche calcolato dall'accessor — doppia fonte di verità

Il model Tournament ha zone_id sia come colonna di database che come attributo appended calcolato dall'accessor. Il commento stesso nel codice dice "Manteniamo per compatibilità". Questo crea ambiguità: qual è il valore corretto?

⚠️
L'accessor getZoneIdAttribute() esegue potenzialmente una query extra ($this->club()->first()) nel fallback. Se chiamato in un loop su una collection di tornei non loaded con club, causa N+1.
❌ Prima — accessor con fallback query potenzialmente N+1
public function getZoneIdAttribute()
{
    if ($this->relationLoaded('club') && $this->club) {
        return $this->club->zone_id;
    }
    if (isset($this->attributes['zone_id'])) {
        return $this->attributes['zone_id'];
    }
    if ($this->club_id) {
        $club = $this->club()->first(); // ← query nascosta!
        return $club ? $club->zone_id : null;
    }
    return null;
}
✅ Dopo — rimuovi colonna ridondante, usa relazione
// Opzione A: Rimuovi zone_id dal DB (migration)
// e usa solo tournament->club->zone_id

// Opzione B (se serve performance):
// Mantieni zone_id ma usa un Observer per sincronizzarlo

// app/Observers/TournamentObserver.php
class TournamentObserver
{
    public function saving(Tournament $tournament): void
    {
        if ($tournament->isDirty('club_id') && $tournament->club_id) {
            $tournament->attributes['zone_id'] =
                Club::find($tournament->club_id)?->zone_id;
        }
    }
}

// Il model diventa semplice:
// zone_id è sempre sincronizzato dall'Observer,
// l'accessor non serve più.
protected $appends = []; // rimuovi 'zone_id'
💡 Perché questa scelta

Un Model Observer è il pattern Laravel per mantenere campi derivati sincronizzati. Intercetta l'evento saving e aggiorna zone_id ogni volta che cambia club_id. Così la colonna DB è sempre in sync, l'accessor è eliminato, e non ci sono query nascoste. Il tradeoff è un'operazione in più al save, ma è trascurabile rispetto all'ambiguità corrente.

public function needsReferees(): bool
{

    // la collection in memoria (zero query)

        return $this->assignments->count()
            < $this->required_referees;

💡 Perché questa scelta

Aggiungere proprietà dinamiche a Model Eloquent è un pattern che "funziona" ma rompe il contratto del model, confonde l'IDE, e crea accoppiamento implicito tra view e model. I readonly class di PHP 8.2 sono perfetti come Data Transfer Objects: immutabili, documentati, type-safe. Il pattern fromAssignment() è uno static factory che centralizza la mappatura.
